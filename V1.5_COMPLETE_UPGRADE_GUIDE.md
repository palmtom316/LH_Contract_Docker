## 升级前必读

### ⚠️ 本次升级的关键问题

**实际遇到的问题**：直接更新代码后，访问合同列表报错 "Database Error"

**根本原因**：
- V1.5 新增 MinIO 对象存储功能
- 数据库需要添加新的存储字段（`contract_file_key`、`contract_file_storage` 等）
- **如果不运行数据库迁移，系统将完全不可用**

**影响范围**：
- ❌ 所有合同功能（上游/下游/管理）完全不可用
- ❌ 文件上传/下载功能失败
- ❌ 财务模块受影响
- ✅ 用户登录不受影响

### 🔴 绝对不能做的事

1. **直接 `git pull` 更新代码** - 会导致系统崩溃
2. **跳过数据库迁移** - 系统将完全不可用
3. **不备份就升级** - 无法回滚
4. **使用 `docker compose down -v`** - 会删除所有数据
5. **在用户使用时升级** - 影响业务

### ✅ 必须做的事

1. **完整备份**（数据库 + 文件 + 配置）
2. **运行数据库迁移** - `docker compose exec backend alembic upgrade head`
3. **验证迁移版本** - 必须是 `v1_5_zero_hour_fix (head)`
4. **测试关键功能** - 合同列表、文件上传下载

### 📊 升级时间估算

- **总时间**: 15-30 分钟
- **停机时间**: 5-10 分钟
- **回滚时间**（如需要）:
  - 使用VM快照: 1-5 分钟 ⚡
  - 使用应用层备份: 10-15 分钟
- **建议时段**: 业务低峰期（如晚上或周末）

```bash

mkdir -p "$BACKUP_DIR"

# 仅备份数据库（最关键）

# 备份配置文件

# 验证备份
ls -lh "$BACKUP_DIR"
echo "✅ 应用层备份完成: $BACKUP_DIR"
echo "$BACKUP_DIR" > .last_backup_path
```

**为什么需要双重备份**：
- VM快照用于灾难性失败的快速恢复

## 🔴 2026-01-13 生产升级经验教训

### 根本原因分析

本次生产升级遇到多个数据库错误，根本原因如下：

| 问题 | 详细说明 |
|------|----------|
| **Git 分支与实际代码不一致** | 生产环境停留在 `release/V1.3` 分支，但代码实际包含了 V1.4.x 的功能修改（约40+个文件的本地修改） |
| **Alembic 迁移状态不明** | 容器内缺少 `alembic.ini`，无法确认当前数据库迁移版本 |
| **手动修改数据库** | 部分 V1.5 的列可能在之前的升级中被手动添加，导致迁移脚本报"列已存在"错误 |
| **迁移脚本不完整** | V1.5 的 Alembic 迁移脚本没有覆盖所有需要新增的表/列 |

### 实际遇到的错误

```
1. DuplicateColumn: column "approval_pdf_key" of relation "zero_hour_labor" already exists
   → 原因：该列在之前的手动修改中已添加，但 Alembic 不知道

2. UndefinedColumnError: column "xxx" does not exist (多个表)
   → 原因：Alembic stamp head 后，系统认为迁移完成，但实际列未添加
```

**受影响的表（共21个）**：
- `contracts_upstream/downstream/management` - 缺少 MinIO 存储列
- `finance_upstream_invoices` - 缺少 `file_key`, `storage_provider`
- `finance_upstream_receivables` - 缺少 `file_key`, `storage_provider`
- `finance_upstream_receipts` - 缺少 `file_key`, `storage_provider`
- `finance_downstream_invoices` - 缺少 `file_key`, `storage_provider`
- `finance_downstream_payables` - 缺少 `file_key`, `storage_provider`
- `finance_downstream_payments` - 缺少 `file_key`, `storage_provider`
- `finance_management_invoices` - 缺少 `file_key`, `storage_provider`
- `finance_management_payables` - 缺少 `file_key`, `storage_provider`
- `finance_management_payments` - 缺少 `file_key`, `storage_provider`
- `downstream_settlements` - 缺少 `file_key`, `storage_provider`
- `management_settlements` - 缺少 `file_key`, `storage_provider`
- `project_settlements` - 缺少多个报告类列
- `expenses_non_contract` - 缺少审批相关列
- `zero_hour_labor` - 缺少文件相关列

## 🔄 升级后验证脚本

```bash
#!/bin/bash
# post_upgrade_verify.sh - V1.5 升级后验证脚本

echo "=== V1.5 升级后验证 ==="

PASS=0
FAIL=0

# 1. Alembic 版本
echo -n "1. Alembic 版本检查: "
ALEMBIC=$(docker compose exec -T backend alembic current 2>/dev/null | grep "head")
if [[ "$ALEMBIC" == *"v1_5_zero_hour_fix"* ]]; then
    echo "✅ $ALEMBIC"
    ((PASS++))
else
    echo "❌ 未达到 v1_5_zero_hour_fix"
    ((FAIL++))
fi

# 2. 服务健康检查
echo -n "2. 后端健康检查: "
HEALTH=$(curl -s http://localhost:8000/health 2>/dev/null)
if [[ "$HEALTH" == *"healthy"* ]]; then
    echo "✅ 健康"
    ((PASS++))
else
    echo "❌ 不健康: $HEALTH"
    ((FAIL++))
fi

# 3. 数据库列检查
echo -n "3. contracts_upstream 新列: "
COLS=$(docker compose exec -T db psql -U lh_admin -d lh_contract_db -t -c \
    "SELECT count(*) FROM information_schema.columns 
     WHERE table_name = 'contracts_upstream' 
     AND column_name IN ('contract_file_key','contract_file_storage','approval_pdf_key','approval_pdf_storage');" 2>/dev/null)
if [[ "$COLS" -eq 4 ]]; then
    echo "✅ 4/4 列存在"
    ((PASS++))
else
    echo "❌ 只有 $COLS/4 列"
    ((FAIL++))
fi

# 4. 容器状态
echo -n "4. 容器运行状态: "
RUNNING=$(docker compose ps --filter "status=running" -q | wc -l)
TOTAL=$(docker compose ps -q | wc -l)
if [[ "$RUNNING" -eq "$TOTAL" ]]; then
    echo "✅ 全部 $TOTAL 个容器运行中"
    ((PASS++))
else
    echo "⚠️  $RUNNING/$TOTAL 个容器运行中"
    ((FAIL++))
fi

# 5. API 测试
echo -n "5. 合同列表 API: "
API=$(curl -s "http://localhost:8000/api/v1/contracts/upstream/?page=1&page_size=1" 2>/dev/null)
if [[ "$API" != *"error"* && "$API" != *"Error"* ]]; then
    echo "✅ 正常"
    ((PASS++))
else
    echo "❌ 错误: ${API:0:50}..."
    ((FAIL++))
fi

echo ""
echo "=== 验证结果 ==="
echo "✅ 通过: $PASS 项"
echo "❌ 失败: $FAIL 项"

if [[ $FAIL -eq 0 ]]; then
    echo "🎉 升级成功！"
else
    echo "⚠️  请检查失败项并修复"
fi
```

## 📝 未来升级建议

1. **规范化版本管理**
   - 生产环境必须使用 release 分支
   - 禁止本地修改代码，所有修改必须通过 Git 提交
   - 每次升级前确保 `git status` 显示 "working tree clean"

2. **完善 Alembic 迁移脚本**
   - 使用 `IF NOT EXISTS` 语法避免重复创建错误
   - 迁移脚本必须覆盖所有需要变更的表

3. **升级前必做**
   - 创建 VM 快照（最重要！）
   - 运行升级前检查脚本
   - 备份数据库

4. **升级后必做**
   - 运行升级后验证脚本
   - 测试所有关键功能
   - 保留 VM 快照至少 7 天
