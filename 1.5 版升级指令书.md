# 工程合同管理系统：架构迁移与智能化升级指令书

> **文件版本**: v1.0
> **适用对象**: AI 辅助编程助手 (Cursor/Windsurf/ChatGPT)
> **目标**: 指导从 Legacy (FastAPI+Vue2/3) 向 Modern (Supabase+TS+MobileFirst) 架构迁移

---

## 1. 项目愿景与目标
本指南旨在指导 AI 助手将原有的 Python (FastAPI) + Vue 3 工程合同管理系统，迁移至以 **Supabase + MinIO + TypeScript** 为核心的现代架构。

**核心原则：** 1. **渐进式迁移**：不推翻重来，新旧共存。
2. **数据零丢失**：文件与数据库记录必须严格保护。
3. **移动端优先**：解决现场办公痛点。
4. **安全左移**：从代码级安全下沉至数据库级安全。

---

## 2. 现状与目标架构对比

| 模块         | 原有技术栈 (Legacy)             | **目标技术栈 (Target)**                      | 迁移策略                                                  |
| :----------- | :------------------------------ | :------------------------------------------- | :-------------------------------------------------------- |
| **后端/DB**  | FastAPI + SQLAlchemy + Postgres | **Supabase (Postgres + Auth + Realtime)**    | 混合架构：Supabase 接管 CRUD，FastAPI 保留用于复杂逻辑/AI |
| **前端**     | Vue 3 (JS) + Element Plus       | **Vue 3 (TS) + Vant UI (移动端) + Tailwind** | 引入 TS，PC/移动端视图分离，逻辑复用                      |
| **文件存储** | 本地文件系统 / NAS              | **MinIO (S3 协议兼容)**                      | 迁移旧文件至 MinIO，建立索引映射                          |
| **安全性**   | 代码层 RBAC (JWT)               | **数据库层 RLS (行级安全) + pgAudit**        | 权限下沉至数据库，启用审计日志                            |
| **智能化**   | 无                              | **pgvector + 本地 LLM (Ollama)**             | 启用向量检索，支持合同语义搜索                            |

## 3. 核心指令：数据安全与文件迁移 (最高优先级)

**⚠️ 警告：在此阶段，AI 必须严格遵守“只读不删”原则，确保生产环境绝对安全。**

### 3.1 数据库记录迁移
1.  **全量备份**：在任何操作前，强制要求编写脚本执行 `pg_dump`，并将备份文件加密上传至异地存储。
2.  **Schema 同步**：使用 Supabase 提供的 Migration 工具同步原有表结构。
3.  **数据导入验证**：编写 Python 脚本进行数据校验（Row Count 对比、关键金额字段 Sum 对比），确保迁移后数据一致。

### 3.2 存量文件迁移方案 (MinIO)
原有系统中上传的合同 PDF/图片是核心资产，必须确保**文件不损坏、路径不丢失**。

* **步骤 A：部署 MinIO**
    * 在 Proxmox 环境中通过 Docker 部署 MinIO。
    * 创建 Bucket：`contracts-archive` (归档旧文件), `contracts-active` (新文件)。
* **步骤 B：编写“非破坏性”迁移脚本 (Python)**
    * 遍历原系统文件存储目录。
    * 计算文件 MD5 哈希值。
    * 使用 `boto3` 或 MinIO SDK 将文件上传至 MinIO。
    * **关键动作**：上传成功后，**不要删除原文件**，而是将原文件路径重命名为 `_migrated_filename`，保留 30 天以备回滚。
* **步骤 C：更新数据库映射**
    * 在数据库中新增字段 `file_key` 和 `storage_provider` (默认为 'local', 迁移后更为 'minio')。
    * 更新记录指向 MinIO 的对象 Key。

## 4. 前端重构指令：TypeScript 与移动端适配

### 4.1 TypeScript 迁移标准
* **禁止使用 AnyScript**：严禁在核心业务逻辑中使用 `any` 类型。
* **类型生成**：必须使用 `supabase gen types typescript` 自动生成数据库类型定义，并保存在 `src/types/database.ts`。
* **Composables 模式**：所有业务逻辑（API 请求、状态管理）必须抽离为 Vue Composables (如 `useContractList.ts`)，确保 PC 端和移动端视图共用同一套逻辑。

### 4.2 移动端适配策略 (Mobile First)
* **视图分治**：
    * 检测到移动设备时，加载 `MobileView.vue`，使用 **Vant UI** 组件（卡片式布局、底部弹窗、下拉刷新）。
    * PC 端保留 **Element Plus** 组件（表格、侧边栏）。
* **样式重构**：引入 **Tailwind CSS**，替换原有的手写 CSS 媒体查询，提高开发效率。

## 5. 后端与安全架构指令

### 5.1 混合后端架构 (Hybrid Backend)
* **Supabase (Direct)**：前端对于简单的读取（列表、详情）和基础写入，直接使用 `supabase-js` SDK 调用，利用 RLS 保证安全。
* **FastAPI (Worker)**：
    * 保留 FastAPI 处理复杂业务（如：合同生成 PDF、Excel 复杂报表导出）。
    * FastAPI 需升级为 **SQLModel**，并配置为 Supabase 的 Service Role (管理员权限) 进行操作。
    * 引入 **pgvector** 逻辑：FastAPI 后台任务负责将合同文本向量化并存回数据库。

### 5.2 安全增强 (RLS & Audit)
* **RLS 策略强制**：所有新建数据表必须开启 RLS (Enable Row Level Security)。
    * 示例策略：`CREATE POLICY "Allow select for own dept" ON contracts FOR SELECT USING (auth.jwt() ->> 'dept_id' = dept_id);`
* **审计日志**：开启 `pgAudit` 扩展或创建 Trigger，记录关键表（合同表、款项表）的 `UPDATE` 和 `DELETE` 操作到 `audit_logs` 表。

## 6. 实施路线图与验收标准

请 AI 助手按照以下顺序执行代码生成任务：

1.  **Phase 1 (基建)**: 部署 MinIO 和 Supabase (Docker Compose)，配置互通。
2.  **Phase 2 (数据)**: 运行文件迁移脚本，验证文件在 MinIO 可预览。
3.  **Phase 3 (后端)**: 配置 Postgres RLS 策略，生成 TS 类型定义。
4.  **Phase 4 (前端)**: 引入 Tailwind 和 Vant UI，重构“合同列表”页面实现 PC/Mobile 视图分离。
5.  **Phase 5 (智能)**: 在数据库开启 pgvector，编写 FastAPI 接口实现语义搜索。

**验收标准：**
1.  旧合同文件能否在移动端流畅打开？
2.  PC 端和移动端能否同时看到最新的合同状态（Realtime 验证）？
3.  数据库强制访问控制（RLS）是否生效？（尝试用非授权账号读取数据应返回空）
4.  **原生产环境数据无任何丢失或损坏。**
