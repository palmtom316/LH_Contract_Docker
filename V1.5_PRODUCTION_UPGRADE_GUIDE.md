# V1.4.1 → V1.5 生产系统升级指引

> **重要提示**：本文档专门针对已部署 V1.4.1 的生产系统升级到 V1.5  
> **预计升级时间**：15-30 分钟  
> **预计停机时间**：5-10 分钟

---

## 📋 升级前必读

### 本次遇到的问题说明

**问题现象**：直接启动 V1.5 代码后，访问上游合同列表报错"Database Error"

**根本原因**：
- V1.5 新增了 MinIO 对象存储功能
- 数据库需要添加新的存储字段（`contract_file_key`、`contract_file_storage` 等）
- 如果不运行数据库迁移，代码会查询不存在的列，导致系统崩溃

**影响范围**：
- ❌ 所有合同功能（上游/下游/管理）完全不可用
- ❌ 文件上传/下载功能失败
- ❌ 财务模块受影响
- ✅ 用户登录不受影响

**结论**：**绝对不能**直接 `git pull` 更新代码！必须按照本指引执行升级。

## ✅ 正确的升级流程

### 方式一：使用自动化升级脚本（强烈推荐）

项目已提供完整的升级脚本，会自动处理所有步骤。

#### 1. 升级前准备

```bash
# 进入项目目录
cd /path/to/LH_Contract_Docker

# 检查当前版本（应该是 V1.4.1）
grep VERSION backend/app/core/config.py

# 检查磁盘空间（需要至少 2 倍当前数据大小）
df -h .

# 检查当前数据量
du -sh backend/uploads
docker-compose exec db psql -U lh_admin -d lh_contract_db -c "SELECT pg_size_pretty(pg_database_size('lh_contract_db'));"
```

#### 2. 手动备份（额外保险）

```bash
# 创建备份目录
mkdir -p ./manual_backup_$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="./manual_backup_$(date +%Y%m%d_%H%M%S)"

# 备份数据库
docker-compose exec -T db pg_dump -U lh_admin lh_contract_db > "$BACKUP_DIR/database.sql"

# 备份文件
cp -r ./backend/uploads "$BACKUP_DIR/uploads_backup"

# 备份配置
cp .env "$BACKUP_DIR/.env.backup"
cp docker-compose.yml "$BACKUP_DIR/docker-compose.yml.backup"

# 验证备份
ls -lh "$BACKUP_DIR"
echo "✅ 备份完成: $BACKUP_DIR"
```

#### 3. 执行自动升级

```bash
# 拉取最新代码
git fetch origin
git checkout main
git pull origin main

# 检查升级脚本是否存在
ls -l scripts/production_upgrade_1.4.1_to_1.5.0.sh

# 添加执行权限
chmod +x scripts/production_upgrade_1.4.1_to_1.5.0.sh

# 执行升级（建议在维护窗口执行）
./scripts/production_upgrade_1.4.1_to_1.5.0.sh
```

#### 4. 升级过程中的选择

脚本会询问文件迁移选项：

```
Choose option (1/2/3):
  1) Keep files local (no migration) - RECOMMENDED
  2) Migrate files to MinIO
  3) Run dry-run to preview migration
```

**建议选择 1**：保持文件在本地存储
- ✅ 零风险，升级快速完成
- ✅ 系统立即可用
- ✅ 后续可以随时迁移到 MinIO

#### 5. 验证升级成功

脚本会自动验证，你也可以手动检查：

```bash

docker-compose ps
# 所有服务应该显示 "Up" 状态

# 检查数据库迁移版本
docker-compose exec backend alembic current
# 应该显示: v1_5_zero_hour_fields (head)

# 测试后端 API
curl http://localhost:8000/docs
# 应该返回 Swagger 文档页面

# 检查后端日志（确认无错误）
docker-compose logs --tail=50 backend | grep -i error
```

### 方式二：手动升级（仅在脚本失败时使用）

如果自动脚本遇到问题，可以手动执行关键步骤：

#### 步骤 1：备份（同上）

#### 步骤 2：停止服务

```bash
docker-compose down
```

#### 步骤 3：更新代码

```bash
git pull origin main
```

#### 步骤 4：更新配置文件

编辑 `.env`，确保包含以下配置：

```bash
# Environment
ENV=production
DEBUG=false

# MinIO Configuration (如果不存在则添加)
MINIO_ROOT_USER=minioadmin
MINIO_ROOT_PASSWORD=minioadmin123  # 生产环境请改为强密码
MINIO_ENDPOINT=minio:9000
MINIO_ACCESS_KEY=${MINIO_ROOT_USER}
MINIO_SECRET_KEY=${MINIO_ROOT_PASSWORD}
MINIO_SECURE=false
MINIO_BUCKET_ACTIVE=contracts-active
MINIO_BUCKET_ARCHIVE=contracts-archive
```

#### 步骤 5：启动数据库并运行迁移（关键！）

```bash
# 启动数据库和 Redis
docker-compose up -d db redis
sleep 10

# 重新构建后端（包含 MinIO 依赖）
docker-compose build --no-cache backend

# 启动后端
docker-compose up -d backend
sleep 10

# 运行数据库迁移（最关键的步骤！）
docker-compose exec backend alembic upgrade head

# 验证迁移成功
docker-compose exec backend alembic current
# 应该显示: v1_5_zero_hour_fields (head)
```

#### 步骤 6：启动 MinIO 并创建存储桶

```bash
# 启动 MinIO
docker-compose up -d minio
sleep 10

# 创建存储桶
docker-compose exec backend python -c "
from minio import Minio
import os

client = Minio(
    'minio:9000',
    access_key=os.getenv('MINIO_ROOT_USER', 'minioadmin'),
    secret_key=os.getenv('MINIO_ROOT_PASSWORD', 'minioadmin123'),
    secure=False
)

for bucket in ['contracts-active', 'contracts-archive']:
    if not client.bucket_exists(bucket):
        client.make_bucket(bucket)
        print(f'✅ Created bucket: {bucket}')
    else:
        print(f'✅ Bucket already exists: {bucket}')
"
```

#### 步骤 7：重建前端并启动所有服务

```bash
# 重建前端
docker-compose build --no-cache frontend

# 启动所有服务
docker-compose up -d

# 检查服务状态
docker-compose ps
```

## 🚨 如果升级失败怎么办？

### 快速回滚步骤

```bash
# 1. 停止所有服务
docker-compose down

# 2. 恢复数据库
BACKUP_DIR="./manual_backup_20260112_140000"  # 替换为你的备份目录
docker-compose up -d db
sleep 5
docker-compose exec -T db psql -U lh_admin -d lh_contract_db < "$BACKUP_DIR/database.sql"

# 3. 恢复文件
rm -rf ./backend/uploads
cp -r "$BACKUP_DIR/uploads_backup" ./backend/uploads

# 4. 恢复配置
cp "$BACKUP_DIR/.env.backup" .env
cp "$BACKUP_DIR/docker-compose.yml.backup" docker-compose.yml

# 5. 回退代码到 V1.4.1
git checkout v1.4.1  # 或者具体的 commit hash

# 6. 重启服务
docker-compose up -d

# 7. 验证回滚
docker-compose ps
curl http://localhost:8000/docs
```

## 📌 常见问题

### Q1: 升级过程中断了怎么办？

**A**: 查看升级日志文件，定位错误原因：
- 如果是网络问题，重新运行升级脚本
- 如果是配置问题，修正后重新运行
- 如果不确定，执行回滚操作

### Q2: 数据库迁移报错 "relation already exists"

**A**: 可能是迁移状态不一致，检查并修复：

```bash
# 查看迁移历史
docker-compose exec backend alembic history

# 查看当前版本
docker-compose exec backend alembic current

# 如果需要，标记为已迁移（不实际运行 SQL）
docker-compose exec backend alembic stamp head
```

### Q3: 前端显示空白页面

**A**: 检查前端容器和日志：

```bash
# 检查前端容器状态
docker-compose ps frontend

# 查看前端日志
docker-compose logs --tail=50 frontend

# 重建前端
docker-compose build --no-cache frontend
docker-compose up -d frontend
```

### Q4: 后端 API 返回 500 错误

**A**: 这通常是数据库迁移未完成导致的：

```bash
# 检查迁移状态
docker-compose exec backend alembic current

# 如果不是 v1_5_zero_hour_fields，运行迁移
docker-compose exec backend alembic upgrade head

# 重启后端
docker-compose restart backend

# 查看详细错误日志
docker-compose logs --tail=100 backend
```

### Q5: MinIO 无法访问

**A**: 检查 MinIO 服务：

```bash
# 检查 MinIO 容器状态
docker-compose ps minio

# 查看 MinIO 日志
docker-compose logs minio

# 重启 MinIO
docker-compose restart minio

# 测试 MinIO API
curl http://localhost:9000/minio/health/live
```
