## 整改优先级

### 第一阶段：立即整改

#### 1. 修复 `AppPageHeader` 回归

目标：

- 无 `actions` 插槽时仍应渲染标题
- 明确 `title`、`actions`、`description`、`meta` 的最终设计契约

建议动作：

- 去掉整块页头对 `actions` 的依赖
- 若 `description`、`meta` 已决定废弃，则同步删掉 props、调用方与相关测试
- 若仍保留，则恢复一致渲染和断言

产出标准：

- 相关页面标题恢复
- `AppPageHeader` 单测通过
- 共享页头 API 收敛为单一真相

#### 2. 修复通知持久化串用户问题

目标：

- 通知读/删状态按用户隔离
- 退出登录后不会污染下一个用户会话

建议动作：

- 为通知 key 加用户维度命名空间，例如基于用户 ID 或用户名
- 在用户切换、登出、登录恢复时明确初始化和迁移策略
- 为跨用户场景补充测试

产出标准：

- 同浏览器切换账号不共享通知状态
- 本地通知不会跨用户残留

#### 3. 统一客户端 IP 解析逻辑

目标：

- 审计与限流使用同一套“可信代理”规则

建议动作：

- 删除 `audit_service.py` 中独立的宽松实现
- 复用 `backend/app/core/rate_limit.py` 的可信代理逻辑，或抽到公共模块
- 补充单元测试覆盖以下情况：
  - 直连请求
  - 可信代理转发
  - 非可信代理伪造转发头

产出标准：

- 审计记录与限流对同一请求得出一致 IP

### 第二阶段：测试恢复

#### 4. 恢复前端测试绿灯

目标：

- `npm test --prefix frontend` 全绿

建议动作：

- 区分两类失败：
  - 真回归：修源码
  - 旧断言：修测试
- 对共享 UI 组件建立更清晰的契约型测试，避免测试只盯样式字面量

产出标准：

- 当前 2 个失败测试清零
- 共享组件测试更关注行为与结构，不只关注单行样式文本

#### 5. 重构后端测试分层，降低对本地 PostgreSQL 的硬依赖

目标：

- 单元测试可在无本地 PostgreSQL 的环境下运行
- 只有集成测试才要求真实数据库

建议动作：

- 把现有测试分成：
  - 纯单元测试
  - 服务层数据库测试
  - API 集成测试
- 用 marker 或目录分层执行
- 为数据库测试提供明确启动方式或测试容器方案
- 对无法脱离 PostgreSQL 的测试，增加环境探测与跳过说明

产出标准：

- 最小测试集可本地一键执行
- 集成测试依赖条件清晰

#### 6. 修正错误处理契约与测试断言

目标：

- `AppException` 的结构有且只有一套标准

建议动作：

- 明确规范：
  - `HTTPException.detail` 是否永远为结构化对象
  - 前端错误读取逻辑如何消费
- 更新 `backend/tests/test_errors.py`
- 补充 API 层面的错误响应断言，避免只测 Python 对象属性

产出标准：

- 错误模型、后端测试、前端消费方式一致

## 风险与注意事项

- UI 共享组件整改会牵动多个页面，必须先补测试或同步更新测试
- 通知 key 命名空间调整涉及本地旧数据迁移，需考虑向后兼容
- 审计 IP 逻辑修改后，线上反代配置需同步核对 `TRUSTED_PROXIES`
- 清理仓库跟踪文件时要避免误删用户实际需要的模板资源
